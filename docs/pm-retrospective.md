## 1. 我做了什么

一个可评测的迷你 agent harness:用受控消融,量化 compaction / 权限闸门 / sub-agent 三种可靠性控制,如何改变一个深度研究 agent 的**任务成功率、成本、失败模式**。两个北极星:EGTSR 与 cost-per-success。

## 3. 做对的 / 做错的(诚实)

**做对的**
- **失败分类学**从一开始就立(infra 失败不计为 agent 失败),保证了 EGTSR 公平,且 C0 真 run 时直接复用。
- **保留失败 artifact**:G3/G4 各挂过一次,我把失败件留作证据而非悄悄重跑——门对自己也 fail-closed。
- **拿到 1.0 没当成绩**:主动标注它是两任务 smoke、非 primary headline,守住了"机器正确 ≠ 模型能力"。

**做错的 / 会改的**
- **真实 run 跑得偏晚**:4 个测量 bug(markdown 清洗、rubric 假阴性、引用排版、trace 版本)只有真 run 才冲得出来。这套"先把基建做扎实再跑真 run"对**可靠性实验室**是对的,但在**产品语境**下,我会更早跑一次粗糙真 run,把测量 bug 提前逼出来。
- **evidence_id 用了 UUID**:设计 schema 时凭直觉用了 UUID,读 Anthropic《Writing Effective Tools》才发现它恰好是"降低 retrieval 精度"的反模式——说明我应该**先读透 tool design 再定 schema**,而不是边做边补。已记入 backlog(改 0-indexed 别名)。
- **过度建造的拐点识别偏晚**:基建端到端跑通后,继续往 80-run 矩阵推的**边际 portfolio 价值在递减**,而打包 + 投递的边际价值更高。一个更果断的 PM 会更早从"建"切到"讲 + 投"。

## 4. 关于做 AI 产品,我学到的(可迁移)

1. **在评测里,测量本身就是产品**——修那把歪尺子,比让 agent 跑通更难也更值钱。
2. **cost-per-success = agent 的单位经济学**——按结果定价时,可靠性就是毛利;这是比"准确率"更锋利的产品指标。
3. **trust but verify,连工具和顾问的建议也要核**——我曾被建议用某低价中转(七牛/灵眸),我回官方文档核实后推翻了,避免了污染整个评测。判断不能外包给任何工具或人。
4. **机器正确 ≠ 能力**——绿色数字第一反应该是"管道对了吗",不是"模型牛"。

## 5. 现状 + 下一步 + 我的判断

**现状**:provider 资格全绿 → trace/evidence 契约 → 冻结源桥接 → C0 → 任务契约 + preflight → **第一个真实 EGTSR**。架构层端到端跑通且全程可复现。

**已显式延后**:judge、C1–C3 配置、80-run 矩阵、S1 记忆横评。这些是 more-of-the-same,不再有新架构风险。

**我的判断**:这是一个该从"继续建"切到"打包 + 投递"的拐点。项目现在已经能讲——一个真实、可复现、有成本核算、有失败分类的 agent reliability lab。继续做到完美不是目标;**用它证明我理解并能解决 agent 落地的核心问题,才是。**
