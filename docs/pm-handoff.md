## 0. 一句话产品定义

> 一个**可评测的迷你 agent harness**：用受控消融实验，量化 compaction、权限闸门、sub-agent 三种可靠性控制，分别如何改变一个深度研究 agent 的**任务成功率、成本、失败模式**。

它**不是** Claude Code 的复刻，**不是**一个 SaaS 产品。它是一个回答"什么让 agent 在生产级场景里可靠"的受控实验。

## 1. 责任边界：哪些是 PM 定义，哪些是研发实现

这是本文最重要的一张表。**PM 定义"契约与判据"，研发选择"数据结构与实现"。**

| 维度 | PM 定义（我负责） | 研发实现（你负责） |
|---|---|---|
| 产品是什么 / 不是什么 | scope 边界、portfolio 主张、目标用户与 JTBD | — |
| 成功的定义 | 研究问题、北极星指标、各项通过阈值 | 指标怎么算、聚合逻辑 |
| 实验设计 | C0–C4 消融阶梯、每阶测什么产品决策 | loop、adapter、工具桥接的代码 |
| 数据可信度 | Provider 资格判据（G1–G4）、供应商排除、单模型策略 | probe 脚本、断言实现 |
| 可复现性 | 冻结快照政策、禁止静默换模型、版权边界 | hash 校验、manifest、缓存机制 |
| 公平计分 | 失败分类学（什么算 agent 失败 vs 基础设施失败） | status 标签的落地、trace 不变量 |
| 安全策略 | 权限模型 allow/ask/deny、不可逆动作边界 | 权限引擎、审批流 |
| 交付标准 | MVP-1/2/3 的验收口径 | 满足验收的具体功能 |
| 风险治理 | 风险登记与对策方向 | 对策的技术落地 |

**判断口诀**：如果一个问题的答案需要"业务/领域/价值/公平/合规"判断 → 是 PM 的；如果只需要"用什么库、什么数据结构、怎么写得更快" → 是研发的。

## 2. 已锁定的产品决策（研发的单一事实来源）

以下是**已经定了、研发可以直接依赖**的边界。改动需走 PM 评审。

### 2.1 范围（In / Out of Scope）
- **In**：单条深度研究工作流（规划→检索→阅读→记录证据→综合→带引用报告）；4 个累积 harness 配置；冻结且版本化的语料；20 个研究任务；结构化 trace / checkpoint / 失败标签；自动评分 + 小样本盲审；**单模型单供应商**做主消融。
- **Out（14 天 MVP 不做）**：把 Mem0/Letta/Zep/Cognee 当记忆后端比较（那是后续 S1）；多模型 benchmark；生产级鉴权 / 真实破坏性动作；通用浏览器自动化；SaaS 界面；声称超出本任务集的统计普适性。

### 2.2 研究问题（= 成功的定义）
> 哪些 harness 控制能**实质性提升**证据化任务成功率，而**不会**让成功的运行变得不成比例地昂贵？

### 2.3 演示问题（对外 demo 用）
> 比较 Mem0、Letta、Zep、Cognee 的 agent 记忆架构及其取舍。

### 2.4 北极星与指标体系
- **北极星 1**：证据化任务成功率（EGTSR）——只有全部硬门槛通过才算成功。
- **北极星 2**：每个成功任务的成本（cost per successful task）。
- **支撑指标**：引用精确率、必需声明覆盖率、单次调用峰值上下文 token、中断恢复成功率。
- **诊断指标**：延迟、token、压缩比、压缩后关键事实保留率、审批次数、权限误判、sub-agent 交接体积、重复源、工具调用错误、无支撑声明数。
- **PM 决策点**：每个硬门槛的**阈值**由 PM 定（见 §4 未决项）。

### 2.5 实验配置（消融阶梯）
| ID | 配置 | 测的产品决策 |
|---|---|---|
| C0 | ReAct baseline | 没有 harness 控制时会怎么失败？ |
| C1 | +compaction | 能否在不丢答案关键证据的前提下压上下文增长？ |
| C2 | +permission gate | 能否以可接受的摩擦挡住不安全/越界动作？ |
| C3 | +sub-agents | 委派带来的覆盖率提升是否值回开销？ |

### 2.6 供应商资格与可信度（治理决策）
- **主模型**：`deepseek-v4-flash`，单模型单供应商（DeepSeek 官方 API）。
- **外部效度子集**：Claude Sonnet 4.6 经 AiHubMix，独立报告的 32-run 子集，须先过资格门。
- **排除**：302.ai——其文档自承风控期可能静默切到 GLM/K2，违反固定模型控制。
- **资格门 G1–G4**（全部已 PASS）：身份 / 缓存核账 / 工具保真 / 稳定性 soak。**任一不过的供应商不得产出计分数据。**

### 2.7 可复现性与版权边界
- 冻结带日期的源快照 + content hash，**fail-closed 校验**（hash 不符即拒绝，不告警放行）。
- 公开 manifest 可校验完整性，但**不是第三方全文的自包含档案**——全文存本地 gitignored 缓存，避免转载第三方版权内容。

### 2.8 失败分类学（公平计分的根）
计分前先分类，**基础设施失败不得记为 agent 失败**：
`eval_valid` / `agent_failed` / `policy_blocked_expected` / `infra_api_failed` / `source_unavailable` / `judge_required`。

### 2.9 权限模型（C2 用）
读冻结源 / 搜冻结索引 / 写运行产物目录 = Allow；访问非白名单域 / 执行代码 = Ask；读凭证密钥 / 发送·发布·购买·删除·改外部状态 = Deny。权限判定要能区分：正确 allow / 正确 ask / 正确 deny / 过度拦截 / 拦截不足。

### 2.10 预算与终止
即便 baseline，每次运行都受 iteration / model-call / tool-call / token / cost / duration 上限约束，**fail-closed**；预算耗尽记为 agent 失败 + 保留完整 trace（不是崩溃，不是 infra 失败）。

## 4. 仍需 PM 拍板的未决项（研发被它阻塞）

这是交接的核心"asks"。**这些都不是研发能替你决定的——它们是业务/领域/公平判断。**

| # | 未决项 | 为什么是 PM 的 | 阻塞了什么 |
|---|---|---|---|
| 1 | **5 个 benchmark 任务 + rubric**（required_claims / acceptable_sources / known_distractors / comparison_dimensions / citation 期望 / 评分细则） | "正确答案长什么样、及格线在哪"是领域判断，研发无法发明 | 第一个真 EGTSR、MVP-1 |
| 2 | **各指标通过阈值**（必需声明覆盖率？事实正确率？引用精确率/蕴含？） | "多严算成功"是产品判断；阈值定生死 | EGTSR 是否可信 |
| 3 | **冻结哪些源**（哪些是 official primary source） | 源权威性判断 + 触碰 live web | 语料冻结 |
| 4 | **定价记录**（versioned 成本模型） | 成本核算口径是 PM 决策 | cost-per-success 真实值 |
| 5 | **C2 权限策略细节**（每个动作最终 ask/deny） | 安全边界是产品策略 | C2 实现 |
| 6 | **盲审裁决政策**（≥8 run 谁审、分歧如何公布） | 评测公正性治理 | MVP-2 人工审计 |
| 7 | **"成功恢复"判据**（恢复后必须保住什么才算过） | 定义"可靠"的含义 | 恢复测试计分 |

**建议下一步处理顺序**（因为任务与语料耦合）：
设计任务（#1）→ 据任务冻最小语料（#3）→ 对照冻好的文本敲定阈值与 distractor（#2）→ 补定价（#4）→ C0 真跑 → 第一个真 EGTSR。

## 5. 验收口径（MVP gates · PM 定义"什么算完成"）

- **MVP-1 可演示**：C0 端到端跑通一个任务；5 个 benchmark 任务可运行；每个 model/tool 调用进 trace；第一个 EGTSR 可单命令复现；打 demo tag。
- **MVP-2 可对比**：C0–C3 同一接口跑通；20×4 矩阵完成或有明确 status 标注；两个北极星 + 四个支撑指标产出；至少一个中断恢复测试通过；产出结果表 + 失败分类。
- **MVP-3 可发布**：干净环境可复现；3 分钟 demo；中英文文章（讲决策与失败，不只讲功能）；一页 PM 复盘；README 含架构、结果、局限、与 V2 的链接。

## 6. 风险与治理边界（PM 持有）

| 风险 | PM 设定的对策方向 |
|---|---|
| 供应商静默换模型 | 每次运行强制 G1 身份核验；不符即作废该运行组 |
| 评测被 LLM judge 一手包办 | 优先确定性检查；公布人工分歧率 |
| 累积配置掩盖单组件效应 | 保留 feature flag；对异常结果做定点回滚 |
| 语料随 live web 漂移 | 冻结带日期快照 + hash |
| scope 膨胀进 S1 记忆比较 | 把记忆产品当研究对象，不当集成后端 |
| 80 次运行超预算 | 先每格一次，分层子集才加重复 |

## 7. PM / 研发 协作契约（往后怎么配合）

- **PM 交付给研发的**：任务 + rubric、阈值、源清单、定价、权限策略、验收口径——即本文 §4 的输出。
- **研发交付给 PM 的**：可单命令复现的 EGTSR、结果表、失败分类、trace 可审计性——以及任何"实现层发现的、影响产品判断的事实"（如某指标无法稳定测量）。
- **共同纪律**：trust but verify——任何"听起来很对"的结论，先核实 primary source 再采信；任何绿灯数字，先分清"机器正确性"还是"能力信号"。

## 附：本文与代码的关系

本文定义**契约**；代码实现契约。当两者冲突时：
- 若是"判据/边界/阈值"冲突 → 以本文为准，研发按本文改实现。
